# 09. Long-term Data Strategy (장기 데이터 전략)

## 개요

6개월 이상 축적된 종단(Longitudinal) 피부 데이터는 NIA의 핵심 자산이다. 이 문서는 장기 데이터를 활용하여 개인 피부 패턴 발견, 계절별 예측, 최적 제품 추천, 커뮤니티 인사이트, 성분 효과 DB 구축, 예측 모델, 그리고 이를 안전하게 보호하는 프라이버시 전략을 정의한다.

## 1. 개인 피부 패턴 분석 (Personal Skin Pattern Analysis)

### 1.1 분석 가능한 패턴 유형

| 패턴 유형 | 최소 데이터 기간 | 분석 내용 |
|----------|----------------|----------|
| **주간 패턴 (Weekly)** | 4주+ | 요일별 피부 컨디션 변동 (예: 주말 음주 후 월요일 트러블) |
| **월간 패턴 (Monthly)** | 3개월+ | 생리 주기와 피부 변화 상관관계, 월초/월말 스트레스 영향 |
| **계절 패턴 (Seasonal)** | 12개월+ | 계절별 피부 타입 변화 (예: 여름 지성 → 겨울 건성) |
| **제품 반응 패턴** | 6개월+ | 개인의 성분별 반응 프로필, 효과 나타나기까지의 평균 시간 |
| **노화 패턴** | 24개월+ | 개인 피부 노화 속도, 주름/탄력 변화 추이 |

### 1.3 패턴 발견 알고리즘

```python
# 주간 패턴 발견 (예시)
def discover_weekly_pattern(daily_scores, min_weeks=4):
    """최소 4주 데이터로 요일별 피부 패턴 분석"""

    # 요일별 그룹핑
    by_weekday = group_by_weekday(daily_scores)

    # ANOVA 검정: 요일 간 유의미한 차이가 있는지
    f_stat, p_value = scipy.stats.f_oneway(*by_weekday.values())

    if p_value < 0.05:
        # 유의미한 차이 존재
        best_day = max(by_weekday, key=lambda d: np.mean(by_weekday[d]))
        worst_day = min(by_weekday, key=lambda d: np.mean(by_weekday[d]))
        return WeeklyPattern(
            significant=True,
            best_day=best_day,
            worst_day=worst_day,
            pattern_strength=f_stat
        )
    return WeeklyPattern(significant=False)
```

## 2. 계절별 피부 변화 분석 및 예측 (Seasonal Skin Analysis)

### 2.1 계절별 피부 변화 요인

| 계절 | 주요 환경 변화 | 예상 피부 영향 |
|------|--------------|---------------|
| **봄 (3-5월)** | 꽃가루, 황사, 자외선 증가, 큰 일교차 | 민감도 증가, 알레르기 반응, 피부 장벽 약화 |
| **여름 (6-8월)** | 고온다습, 강한 자외선, 에어컨 건조 | 피지 증가, 모공 확대, 자외선 손상, T존/U존 편차 확대 |
| **가을 (9-11월)** | 건조, 일교차, 자외선 감소 | 여름 손상 회복기, 각질 증가, 수분 감소 시작 |
| **겨울 (12-2월)** | 극건조, 저온, 실내 난방, 낮은 UV | 극심한 건조, 피부 장벽 손상, 주름 심화, 혈색 저하 |

## 3. 개인 최적 화장품 추천 (Data-driven Top-N Recommendation)

### 3.1 추천 알고리즘 아키텍처

```
                    ┌──────────────────────┐
                    │   추천 엔진 (Top-N)    │
                    └──────────┬───────────┘
                               │
            ┌──────────────────┼──────────────────┐
            ▼                  ▼                  ▼
   ┌─────────────────┐ ┌────────────────┐ ┌──────────────────┐
   │ Content-Based    │ │ Collaborative  │ │ Causal-Based     │
   │ Filtering        │ │ Filtering      │ │ Filtering        │
   │ (성분 기반)       │ │ (유사 유저 기반)│ │ (인과관계 기반)    │
   └────────┬────────┘ └───────┬────────┘ └────────┬─────────┘
            │                  │                    │
            └──────────────────┼────────────────────┘
                               ▼
                    ┌──────────────────────┐
                    │  Hybrid Ranking       │
                    │  (가중 앙상블)          │
                    └──────────┬───────────┘
                               ▼
                    ┌──────────────────────┐
                    │  개인화 Top-N 추천     │
                    └──────────────────────┘
```

### 3.2 추천 방법론 상세

#### (1) Content-Based Filtering (성분 기반)
- 유저에게 효과적이었던 성분 프로필 분석
- 동일/유사 성분 조합을 가진 미사용 제품 추천
- 유저의 "주의 성분"을 포함한 제품 필터링(제외)

#### (2) Collaborative Filtering (유사 유저 기반)
```python
def find_similar_users(target_user, all_users, top_k=50):
    """피부 데이터 기반 유사 유저 탐색"""
    similarities = []

    for user in all_users:
        sim = compute_skin_similarity(
            target_user.skin_profile,
            user.skin_profile,
            features=[
                'skin_type',         # 피부 타입
                'sensitivity',       # 민감도
                'concern_areas',     # 고민 부위
                'age_group',         # 연령대
                'climate_zone',      # 기후대
                'baseline_scores'    # 기준 피부 점수
            ]
        )
        similarities.append((user, sim))

    return sorted(similarities, key=lambda x: x[1], reverse=True)[:top_k]
```

#### (3) Causal-Based Filtering (인과관계 기반) -- NIA 고유 방법론
- 일반 협업 필터링과 달리, **유사 유저에게서 실제 인과관계가 확인된 제품**만 추천
- 단순 상관관계("이 제품을 쓴 사람이 피부가 좋다")가 아닌, 인과관계("이 제품을 쓰기 시작한 후 피부가 좋아졌다")를 기반으로 함
- Difference-in-Differences (DID), Interrupted Time Series (ITS) 등 인과 추론 기법 적용

### 3.3 추천 결과 표시 형식

```
┌─────────────────────────────────────────────────┐
│  당신을 위한 추천 #1                              │
│                                                  │

│                                                  │
```

### 3.4 추천 품질 지표

| 지표 | 정의 | 목표값 |
|------|------|--------|
| **Hit Rate@5** | 추천 Top 5 중 유저가 실제 효과를 본 비율 | 65%+ |
| **NDCG@10** | 추천 순위 품질 (높을수록 좋은 제품이 상위에) | 0.70+ |

| **Serendipity** | 유저가 예상치 못한 효과적 제품 발견 비율 | 20%+ |

## 4. 커뮤니티 데이터 활용 (Community Data Utilization)

### 4.1 익명화 원칙

모든 커뮤니티 데이터는 **개인 식별 불가능(De-identified)** 상태로만 활용:

| 데이터 유형 | 익명화 방법 | 공유 범위 |
|------------|-----------|----------|
| 피부 측정 수치 | K-Anonymity (k≥50), 수치 구간화 | 통계 집계만 공개 |
| 제품 효과 데이터 | 개인 연결 해제, 그룹 통계만 | 제품별 효과 통계로만 |
| 피부 사진 | **절대 공유 금지** | 개인 기기 내 저장만 |
| 피부 프로필 | 일반화(Generalization) | 피부 타입 클러스터로만 |
| 루틴 데이터 | 개인 연결 해제 | 제품 조합 통계로만 |

### 4.2 커뮤니티 인사이트 유형

#### (1) 제품 효과 리포트 (Product Efficacy Report)
```

  모공:    ■■■■■■■■░░  -13.2% (p<0.01)
  피부결:  ■■■■■■░░░░  +8.7%  (p<0.01)
  색소:    ■■■■░░░░░░  -5.1%  (p<0.05)
  수분:    ■■■░░░░░░░  +3.2%  (p=0.12, 유의하지 않음)

피부 타입별 효과:
  지성:  효과 높음 (78% 개선)
  건성:  효과 중간 (52% 개선)
  민감:  주의 (23%에서 자극 반응)

```

#### (2) 루틴 인사이트 (Routine Insight)
- "나와 비슷한 피부 타입이 가장 많이 사용하는 루틴 조합"
- "이 제품과 함께 사용하면 시너지가 좋은 제품" (실제 데이터 기반)
- "이 제품과 함께 사용하면 효과가 감소하는 제품" (길항 작용 감지)

#### (3) 트렌드 인사이트 (Trend Insight)
- "이번 달 가장 효과가 검증된 신제품 TOP 10"
- "20대 복합성 피부에서 올해 가장 효과적인 세럼 TOP 5"
- 지역별/연령별/피부 타입별 트렌드 분석

## 5. 성분 효과 데이터베이스 구축 (Crowdsourced Ingredient Efficacy DB)

### 5.1 데이터베이스 구조

```
Ingredient Efficacy DB
├── 성분 (Ingredient)
│   ├── 기본 정보 (이름, CAS 번호, 카테고리)
│   ├── 효과 프로필 (Efficacy Profile)
│   │   ├── 피부 지표별 효과 (수분, 유분, 모공, 주름, 색소, 피부결, 트러블)
│   │   ├── 효과 크기 (Effect Size) + 신뢰 구간
│   │   ├── 효과 발현 시간 (Time to Effect)
│   │   └── 피부 타입별 효과 차이
│   ├── 상호작용 (Interactions)
│   │   ├── 시너지 성분 (Synergistic)
│   │   ├── 길항 성분 (Antagonistic)
│   │   └── 중립 성분 (Neutral)
│   └── 안전성 (Safety)
│       ├── 자극 발생률 (by 피부 타입)
│       ├── 농도별 자극 임계값
│       └── 장기 사용 안전성 데이터
└── 메타데이터
    ├── 데이터 소스 수 (n)
    ├── 신뢰도 등급
    └── 최종 업데이트 일시
```

### 5.2 데이터 수집 방법론

#### Phase 1: 제품 성분 DB 구축 (0-3개월)
- 한국 식약처(MFDS) 화장품 성분 데이터베이스 연동
- 주요 화장품 성분 사전(INCI Dictionary) 매핑
- 유저가 등록하는 제품의 성분표 자동 파싱 (OCR + 성분 매핑)
- 초기 목표: 상위 500개 성분 커버

#### Phase 2: 효과 데이터 축적 (3-12개월)
- 유저의 제품 사용 데이터와 피부 변화 데이터를 성분 단위로 분해
- 동일 성분을 포함한 서로 다른 제품들의 효과를 Cross-validation
- 최소 n=100 이상의 데이터가 축적된 성분부터 효과 프로필 공개

#### Phase 3: 상호작용 분석 (12-24개월)
- 다중 제품을 동시 사용하는 유저의 데이터로 성분 간 상호작용 분석
- A+B 동시 사용 vs A만 사용 vs B만 사용의 효과 비교
- 최소 n=500 이상의 교차 데이터로 시너지/길항 판정

### 5.3 성분 효과 데이터 신뢰도 관리

| 신뢰도 등급 | 조건 | 표시 |
|------------|------|------|
| **확인됨 (Confirmed)** | n≥500, p<0.01, 3개 이상 제품에서 일관된 결과 | 녹색 체크마크 |
| **유력함 (Probable)** | n≥200, p<0.05, 2개 이상 제품에서 일관된 결과 | 노란색 체크마크 |
| **관찰됨 (Observed)** | n≥50, p<0.10 | 회색 물음표 |
| **데이터 부족 (Insufficient)** | n<50 | 회색 대시 |

### 5.4 B2B 활용: 브랜드를 위한 성분 인사이트

화장품 브랜드에게 판매 가능한 데이터 인사이트:
1. **성분 효과 벤치마크**: "나이아신아마이드 5% vs 10%의 실제 효과 차이"
2. **제형 최적화 인사이트**: "동일 성분이라도 세럼 제형이 크림 제형보다 모공 효과 1.3배"
3. **타겟 고객 프로파일링**: "이 성분에 가장 잘 반응하는 피부 타입 및 연령대"
4. **경쟁 제품 벤치마크**: "동일 카테고리 제품들의 실제 효과 비교 (익명)"

## 6. 예측 모델 (Predictive Models)

### 6.1 모델 개요

| 모델 | 입력 | 출력 | 최소 데이터 요건 | 활용 |
|------|------|------|----------------|------|
| **피부 상태 예측** | 과거 피부 데이터 + 환경 데이터 | 내일/이번 주 피부 상태 예측 | 3개월+ | "내일 피부 컨디션 예보" |
| **제품 효과 예측** | 개인 피부 프로필 + 제품 성분 | 특정 제품의 예상 효과 | 6개월+ 개인 + n≥200 제품 | 구매 전 효과 예측 |
| **트러블 예측** | 과거 트러블 패턴 + 환경/생활 | 트러블 발생 확률 | 4개월+ | "트러블 주의보" 알림 |
| **최적 루틴 예측** | 과거 루틴 + 효과 데이터 | 현재 피부 상태에 최적인 루틴 | 6개월+ | 계절 전환기 루틴 추천 |
| **피부 노화 예측** | 장기 피부 데이터 + 생활 습관 | 1년/3년/5년 후 피부 예측 | 24개월+ | 안티에이징 동기 부여 |

### 6.2 피부 상태 예측 모델 상세

#### 모델 아키텍처
```
[Input Features]
├── 최근 14일 피부 지표 시계열 (7 features × 14 days)
├── 환경 데이터 (기온, 습도, UV 지수, 미세먼지)
├── 생활 데이터 (수면, 음주, 스트레스 레벨)
├── 루틴 데이터 (현재 사용 중인 제품 임베딩)
├── 시간 특성 (요일, 월, 계절, 생리주기 단계)
└── 개인 프로필 (피부 타입, 나이, 성별)
     │
     ▼
[Model: Temporal Fusion Transformer or LSTM]
     │
     ▼
[Output]
├── 내일 피부 점수 예측 (7개 지표)
├── 예측 신뢰 구간 (95% CI)
└── 주요 영향 요인 설명 (Feature Importance)
```

#### 예측 결과 표시 예시
```
[내일의 피부 예보]

종합 컨디션:  ★★★★☆ (78/100)
             ↗ 오늘보다 +3점 예상

지표별 예측:
  수분:   ████████░░  82  (↗+2)
  유분:   ██████░░░░  65  (→ 변동없음)
  모공:   ███████░░░  72  (↗+1)
  트러블: █████████░  88  (↗+5)

주요 영향 요인:
  "오늘 7시간 이상 수면 + 최근 세라마이드 크림 효과"

주의사항:
  "내일 습도 30% 이하 예보. 미스트 휴대를 추천드려요"
```

### 6.3 모델 학습 및 업데이트

| 항목 | 전략 |
|------|------|
| **초기 학습** | 유저 3개월 데이터 축적 후 개인 모델 첫 학습 |
| **업데이트 주기** | 매주 재학습 (Incremental Learning) |
| **Cold Start** | 유사 피부 타입 그룹의 집단 모델로 초기 예측, 데이터 축적 후 개인 모델 전환 |
| **성능 모니터링** | 예측 vs 실제의 MAE(Mean Absolute Error) 추적 |
| **목표 정확도** | 1일 예측: MAE < 5점(100점 만점), 7일 예측: MAE < 10점 |

### 6.4 피부 시뮬레이션 ("What If" 분석)

6개월+ 데이터 유저에게 제공하는 시뮬레이션 기능:

```
[What If 시뮬레이션]

Q: "레티놀 0.3% 세럼을 추가하면?"

예상 결과 (4주 후):
  주름:   -8% (유사 유저 데이터 기반)
  피부결: +5%
  민감도: 첫 2주 일시적 자극 가능 (23% 확률)

권장 사항:
  "격일 사용으로 시작하여 2주 후 매일 사용으로 전환하세요"
  "이 성분은 비타민C와 동시 사용 시 자극이 증가할 수 있어요"
```

## 7. 데이터 프라이버시 및 보안 전략 (Data Privacy & Security)

### 7.1 프라이버시 원칙

1. **최소 수집 (Data Minimization)**: 분석에 필요한 최소한의 데이터만 수집
2. **목적 제한 (Purpose Limitation)**: 수집 시 명시한 목적으로만 사용
3. **사용자 통제 (User Control)**: 데이터 공유/삭제/다운로드의 완전한 통제권 부여
4. **투명성 (Transparency)**: 데이터 활용 방식을 명확하게 공개
5. **보안 우선 (Security First)**: 업계 최고 수준의 보안 적용

### 7.2 데이터 분류 및 보호 수준

| 보호 등급 | 데이터 유형 | 저장 방식 | 접근 권한 |
|----------|-----------|----------|----------|
| **Level 4 (최고)** | 피부 사진 원본 | **기기 내 저장만** (서버 업로드 안 함) | 사용자 본인만 |
| **Level 3 (높음)** | 얼굴 특징 벡터, 피부 분석 상세 결과 | 서버 암호화 저장 (AES-256) | 사용자 + 분석 시스템 |
| **Level 2 (중간)** | 피부 점수 시계열, 제품 사용 기록 | 서버 암호화 저장 | 사용자 + 추천 시스템 |
| **Level 1 (일반)** | 익명화된 통계 데이터 | 분리된 분석 DB | 커뮤니티 기능 + B2B |

### 7.3 핵심 보안 기술 적용

#### (1) 피부 사진 처리
```
[촬영] → [기기 내 AI 분석] → [분석 수치만 서버 전송] → [사진은 기기에만 저장]

* 서버에는 피부 사진 원본이 절대 전송되지 않음
* Edge AI (On-device ML) 활용: Core ML (iOS) / TensorFlow Lite (Android)
* 대안: 사진 전송이 필요한 경우, 처리 즉시 삭제 (Ephemeral Processing)
```

#### (2) 데이터 암호화
| 구간 | 암호화 방식 |
|------|-----------|
| 저장 시 (At Rest) | AES-256-GCM |
| 전송 시 (In Transit) | TLS 1.3 |
| 기기 내 (On Device) | iOS Keychain / Android Keystore |
| 백업 (Backup) | E2E 암호화, 사용자 키 기반 |

#### (3) 접근 제어
- Role-Based Access Control (RBAC)
- 개인 데이터 접근 시 감사 로그(Audit Log) 자동 기록
- 내부 직원도 익명화 데이터만 접근 가능 (원본 접근 불가)

### 7.4 사용자 데이터 권리

| 권리 | 기능 | 구현 |
|------|------|------|
| **열람권** | 내 데이터 전체 조회 | 앱 내 "내 데이터" 메뉴 |
| **다운로드권** | 전체 데이터 다운로드 | JSON/CSV 형식 내보내기 |
| **삭제권** | 전체 또는 특정 기간 데이터 삭제 | 설정 > 데이터 삭제 (30일 내 복구 가능) |
| **공유 통제권** | 커뮤니티 데이터 기여 on/off | 설정 > 데이터 공유 범위 |
| **이동권** | 타 서비스로 데이터 이전 | 표준 포맷 내보내기 |
| **동의 철회권** | 데이터 수집 동의 철회 | 설정 > 개인정보 > 동의 관리 |

### 7.5 규제 준수 (Regulatory Compliance)

| 규제 | 지역 | 준수 전략 |
|------|------|----------|
| **개인정보 보호법 (PIPA)** | 한국 | 개인정보 처리 방침 고지, 동의 획득, 파기 절차 |
| **GDPR** | EU | Data Protection Officer 지정, DPIA 수행, 동의 기반 처리 |
| **CCPA/CPRA** | 미국 (California) | Do Not Sell 옵션, 개인정보 접근/삭제 요청 처리 |
| **생체정보 보호** | 한국/미국 일부 주 | 피부 사진을 생체 정보로 분류 시, 추가 동의 및 보호 조치 |

### 7.6 데이터 보관 정책

| 데이터 유형 | 활성 유저 | 비활성 유저 (6개월+) | 탈퇴 유저 |
|------------|----------|-------------------|----------|
| 피부 사진 (기기 내) | 무제한 | 유저 관리 | 앱 삭제 시 자동 삭제 |
| 분석 결과 | 무제한 | 암호화 보관 | 30일 내 완전 삭제 |
| 루틴 기록 | 무제한 | 암호화 보관 | 30일 내 완전 삭제 |
| 익명화 통계 기여 | 영구 보관 | 영구 보관 | 이미 익명화됨 (삭제 불가) |
| 계정 정보 | 유지 | 12개월 후 휴면 전환 | 즉시 삭제 |
